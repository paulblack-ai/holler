## User Constraints (from CONTEXT.md)

### Locked Decisions

**Tool-use protocol**
- D-01: Tool definitions use OpenAI function calling JSON Schema format as the canonical representation. Each tool (`call`, `sms`, `hangup`, `transfer`) has typed parameters and structured JSON return values. Python-defined with auto-generated JSON Schema export.
- D-02: Compliance blocks return structured error in tool result (`{"status": "blocked", "reason": "..."}`) rather than raising exceptions. The LLM sees the block reason and can inform the human or adjust.
- D-03: Transfer tool issues `uuid_transfer` ESL command to redirect the call to a new destination. Destination goes through compliance gateway first. Attended transfer is v2.
- D-04: Adapter pattern for multi-LLM support. Tool definitions stored in OpenAI format (canonical). Thin adapters translate to/from provider-specific format (Anthropic tool_use, etc.) in the LLM client layer, not the tool layer.

**SMPP integration**
- D-05: Use aiosmpplib (async-native) as SMPP client. Fits established asyncio pattern. Supports message segmentation reassembly and delivery receipt correlation.
- D-06: Persistent SMPP connection with auto-reconnect. Single bind to SMSC initialized at startup alongside Redis and ESL connections.
- D-07: SMS goes through the compliance gateway — TCPA applies to SMS. ComplianceModule gets SMS-aware checking (channel parameter or dedicated method). Same "no bypass" structural guarantee as voice.
- D-08: Delivery receipts handled via SMPP delivery receipt PDU callbacks. Receipts update a delivery status store (Redis or in-memory dict keyed by message ID). Status returned to agent as tool response.

**CLI and onboarding**
- D-09: CLI built with Click framework. `@click.group()` for `holler` command with `init`, `trunk`, `call` subcommands. Entry point registered in pyproject.toml `[project.scripts]`.
- D-10: `holler init` checks GPU availability, downloads faster-whisper + Kokoro ONNX models (with progress bars), generates `.holler.env` config file with defaults, starts Docker Compose services (FreeSWITCH + Redis).
- D-11: Config file `.holler.env` generated by `holler init`. `HollerConfig.from_env()` reads this file first, then environment variables override. Preserves existing from_env() pattern while adding file-based persistence.
- D-12: `holler call +14155551234` uses default voice assistant prompt. `--agent "..."` flag overrides system prompt. Simple, discoverable.

**Agent loop**
- D-13: Holler provides a built-in agent loop. Loop: LLM receives transcript → generates response (possibly with tool calls) → tool calls execute via Holler → results fed back → LLM continues. External systems can also use Holler as a library and run their own loop.
- D-14: VoicePipeline extended to support tool-calling LLM responses. When LLM returns a tool call instead of text, pipeline coordinator executes the tool, feeds result back to LLM, and continues. Tool-call interception is transparent to the voice pipeline.
- D-15: `holler trunk add` configures SIP trunk credentials (host, username, password) and writes them to `.holler.env`. Interactive prompts for required fields, with `--host`, `--user`, `--pass` flags for non-interactive use.

## Summary

Phase 3 adds three interconnected capabilities to Holler: SMPP-based SMS (send, receive, delivery status), an LLM tool-use protocol exposing telecom actions as structured function calls, and a Click-based CLI that completes the four-command onboarding story. All three areas extend existing code patterns rather than introducing architectural departures.

The SMPP layer uses aiosmpplib 0.7.3, which is async-native (asyncio), supports auto-reconnect via `esme.start()` as a long-running task, and handles delivery receipts through a `Hook` subclass. This fits the project's established asyncio pattern exactly. The compliance gateway extends with a `sms_checked()` method parallel to `originate_checked()`, preserving the "no bypass" structural guarantee for SMS as required by D-07.

The tool-use protocol requires two changes to the existing LLM pipeline: (1) add `tools` to chat completion requests in `LLMClient.stream_response()`, and (2) intercept `tool_calls` deltas in the VoicePipeline coordinator. The canonical tool format is OpenAI JSON Schema; Anthropic requires a thin adapter that renames `parameters` → `input_schema`. The adapter lives in the LLM client layer, not the tool layer, matching D-04.

The CLI is a Click `@group` registered as `[project.scripts]` entry point. `holler init` downloads models using `huggingface_hub` and starts Docker Compose services. The `.holler.env` file pattern means config is file-first with env var overrides — a small extension to the existing `HollerConfig.from_env()` dataclass.

**Primary recommendation:** Build in order: (1) SMSClient + compliance extension, (2) tool definitions + LLMClient extension, (3) VoicePipeline tool-call interception, (4) CLI commands. The SMS and tool layers are independent; both feed into the agent loop integration in step 3.

**When to use:** All four tools: `call`, `sms`, `hangup`, `transfer`.

### Pattern 3: Anthropic Adapter (thin translation layer)

**What:** Anthropic uses `input_schema` instead of `parameters`; response uses `tool_use` content blocks instead of `tool_calls` delta; result is sent back as `tool_result` content.

**When to use:** When `LLM_PROVIDER=anthropic` is configured. The adapter is invoked in the LLM client layer.

```python
# OpenAI format → Anthropic format
def openai_to_anthropic(openai_tool: dict) -> dict:
    fn = openai_tool["function"]
    return {
        "name": fn["name"],
        "description": fn["description"],
        "input_schema": fn["parameters"]  # rename only
    }

# Anthropic tool_use response → normalized tool call
def anthropic_response_to_tool_calls(content_blocks: list) -> list:
    return [
        {"id": b["id"], "name": b["name"], "arguments": b["input"]}
        for b in content_blocks if b["type"] == "tool_use"
    ]
```

### Pattern 4: LLMClient Tool-Call Extension

**What:** Extend `stream_response()` to pass `tools` and handle `tool_calls` in the streaming response. When a tool call is detected, yield a sentinel rather than tokens so the pipeline coordinator can intercept it.

**When to use:** When the LLM decides to invoke a tool rather than speak.

```python
# Extension to holler/core/voice/llm.py

                ...
    if tool_calls_accumulator:
```

### Anti-Patterns to Avoid

- **Bypass SMPP through gateway directly:** SMS must go through `ComplianceGateway.sms_checked()`. Same structural guarantee as voice (D-07).
- **Tool definitions scattered across files:** All four tool definitions in one `tools.py`. Adapters in `adapters.py`. Never inline tool schemas in LLM client code.
- **Blocking SMPP in async context:** `smpplib` is synchronous — do not use it. `aiosmpplib` is the project decision (D-05).
- **Delivery status via polling:** Use the `Hook.received()` callback pattern. Never poll for receipt; the SMSC pushes receipts via `deliver_sm`.
- **LLM tool_calls blocking voice pipeline:** When a tool call is detected mid-conversation, the pipeline must emit a "thinking" filler TTS response before executing the tool. Tool execution must not block the audio pipeline for > 100ms without a filler utterance (Pitfall 10 from PITFALLS.md).
- **Click commands calling asyncio.run() directly:** CLI commands must call `asyncio.run(main(...))` once — not nest async calls. Wrap the async `main()` in a sync Click callback.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SMPP keep-alive heartbeats | Custom enquire_link timer | `aiosmpplib` ESME auto-reconnect via `esme.start()` task | ESME.start() handles ENQUIRE_LINK and reconnection; manual keep-alive has timing race conditions |
| SMS message segmentation | Split long messages manually | `aiosmpplib` automatic segmentation | UDH/SAR headers, segment correlation, and reassembly on receive are all handled; manual split breaks delivery receipt correlation |
| Delivery receipt parsing | Parse SMPP receipt text fields | `DeliverSm.parse_receipt()` | Receipt format varies by SMSC vendor; aiosmpplib handles the variations |
| JSON Schema from Python types | Write schemas by hand | Python dataclasses + `pydantic.schema_json()` or manual dict | Maintain one source of truth; schema must be valid for OpenAI strict mode |
| .env file parsing | `open(".holler.env").read()` | `python-dotenv load_dotenv()` | Handles quoting, multiline values, comments; standard library |
| Progress bars for model download | Print statements | `huggingface_hub` built-in tqdm progress | Already provided by `hf_hub_download()`'s `local_files_only` + tqdm |
| Anthropic message format translation | Write full Anthropic client | Thin adapter in `adapters.py` | Anthropic SDK is a large optional dep; adapter keeps Anthropic optional while core uses openai SDK |

**Key insight:** aiosmpplib encapsulates the full SMPP session lifecycle. Trying to manage ENQUIRE_LINK, segmentation, and receipt correlation manually is a multi-week reliability project with edge cases documented across SMPP RFCs. Let the library do it.

## Common Pitfalls

### Pitfall 1: SMPP Delivery Receipt Arrives Days Later

**What goes wrong:** SMSC may deliver receipts seconds or hours after the original message. If the delivery status store is in-memory (dict), a service restart loses all pending receipt correlations. The agent never gets a `delivered` status and the tool call hangs.

**Why it happens:** SMPP receipts are not request-response; they are asynchronous events. In-memory storage works in testing but fails after any restart.

**How to avoid:** Use Redis as the delivery status store (keyed by `message_id` with 48-hour TTL). aiosmpplib's `SimpleCorrelator` can be configured to persist to a file; subclass it to use Redis. For Phase 3, an in-memory dict is acceptable for MVP with a clear upgrade path noted in code comments.

**Warning signs:** Agent receives `sent` but never `delivered` across service restarts.

### Pitfall 2: Tool Call Detected Mid-TTS (Pipeline Deadlock)

**What goes wrong:** VoicePipeline is streaming TTS audio when the LLM response stream delivers a `tool_calls` delta. The pipeline is in `SPEAKING` state. The TTS loop blocks waiting for the token queue, and the tool executor blocks waiting for the pipeline to be idle.

**Why it happens:** Tool calls can interleave with text tokens in streaming responses. The existing `_respond()` in `pipeline.py` handles text tokens only.

**How to avoid:** When `ToolCallSentinel` is received from `token_queue`, the TTS loop must flush and stop (same mechanism as barge-in via `_tts_cancel`), then the coordinator executes the tool, then resumes LLM with tool result. The sentinel-based approach in Pattern 4 avoids adding a second queue.

**Warning signs:** Pipeline freezes during agent loops that involve tool calls; `pipeline.turn_error` logs with timeout errors.

### Pitfall 3: SMPP bind_type Mismatch

**What goes wrong:** SMSC rejects connection with `ESME_RBINDFAIL`. Different SMSC providers require `transceiver`, `transmitter+receiver`, or `transceiver` mode. aiosmpplib defaults to `transceiver` (sends and receives). If the SMSC requires separate TX/RX binds, the single ESME will fail.

**Why it happens:** SMPP 3.4 allows three bind types; not all SMSCs support all three.

**How to avoid:** Document `SMPP_BIND_TYPE` as a config variable. Default to `transceiver`. Add a clear error message if bind fails explaining the bind type options. This is in Claude's Discretion (CONTEXT.md).

**Warning signs:** ESME connects but immediately gets `ESME_RBINDFAIL` — check bind type with SMSC provider documentation.

### Pitfall 4: OpenAI Strict Mode Breaks with Optional Fields

**What goes wrong:** `"strict": true` in tool definitions requires all properties to be in `required` and `additionalProperties: false`. If any parameter is truly optional (e.g., `prompt` in `call()`), strict mode will reject the schema.

**Why it happens:** OpenAI strict mode requires all schema fields to be required. Optional parameters must use a nullable type (`["string", "null"]`) and still be in the `required` array.

**How to avoid:** For optional parameters, use `"type": ["string", "null"]` and include in `required`. This is the OpenAI-recommended pattern for optional fields in strict mode.

**Warning signs:** OpenAI API returns `invalid_request_error` when tools are passed; error message will mention `strict` mode incompatibility.

### Pitfall 5: Click asyncio.run() Nesting

**What goes wrong:** Click command callbacks are synchronous. Calling `asyncio.run()` inside a Click callback that was itself called from within an event loop (e.g., during testing) raises `RuntimeError: This event loop is already running`.

**Why it happens:** pytest-asyncio sets up an event loop for the test; calling `asyncio.run()` from inside a test creates a nested loop.

**How to avoid:** In CLI commands, use `asyncio.run()` only at the top level. In tests, call the async `main()` directly from an async test function. Use `anyio` or `asyncio.get_event_loop().run_until_complete()` pattern where needed.

**Warning signs:** `RuntimeError: This event loop is already running` in tests or during `holler call` invocation from within another async context.

### Pitfall 6: `holler init` Docker Compose Fails Silently

**What goes wrong:** `docker compose up -d` exits zero but FreeSWITCH or Redis fails health checks after start. `holler init` prints "success" but `holler call` immediately fails to connect.

**Why it happens:** Docker Compose returns success when containers start, not when they are healthy. Health check status requires a separate query.

**How to avoid:** After `docker compose up -d`, poll `docker compose ps --format json` or use `docker inspect --format '{{.State.Health.Status}}'` until healthy or timeout. `holler init` must wait for FreeSWITCH ESL port 8021 to accept connections before declaring success.

**Warning signs:** `holler init` prints success, `holler call` immediately fails with `ConnectionRefusedError` on port 8021.

### aiosmpplib Complete SMSClient Skeleton

```python
# Source: https://pypi.org/project/aiosmpplib/ (verified 2026-03-24)
# holler/core/sms/client.py
from __future__ import annotations
import asyncio
from dataclasses import dataclass
from typing import Callable, Optional
import structlog
from aiosmpplib import ESME, PhoneNumber, SubmitSm, AbstractHook, DeliverSm, SmppMessage, Trackable

logger = structlog.get_logger()


@dataclass
class SMSConfig:
    smsc_host: str = "127.0.0.1"
    smsc_port: int = 2775
    system_id: str = "holler"
    password: str = ""
    source_address: str = ""  # Default sender number or short code


class HollerHook(AbstractHook):
    def __init__(self, delivery_store: dict, inbound_handler: Optional[Callable] = None):
        self._store = delivery_store        # {message_id: status}
        self._inbound_handler = inbound_handler

    async def sending(self, smpp_message: SmppMessage, pdu: bytes, client_id: str) -> None:
        logger.debug("sms.sending", log_id=getattr(smpp_message, "log_id", None))

    async def received(self, smpp_message: SmppMessage, pdu: bytes, client_id: str) -> None:
        if isinstance(smpp_message, DeliverSm):
            if smpp_message.is_receipt():
                receipt = smpp_message.parse_receipt()
                stat = receipt.get("stat", "UNKNOWN")
                status_map = {
                    "DELIVRD": "delivered", "EXPIRED": "expired",
                    "UNDELIV": "failed", "REJECTD": "failed",
                    "ACCEPTD": "accepted",
                }
                self._store[smpp_message.log_id] = status_map.get(stat, "unknown")
                logger.info("sms.receipt", log_id=smpp_message.log_id, stat=stat)
            else:
                # Inbound SMS
                sender = str(smpp_message.source)
                text = smpp_message.short_message
                logger.info("sms.inbound", sender=sender, length=len(text))
                if self._inbound_handler:
                    await self._inbound_handler(sender, text)

    async def send_error(self, smpp_message: SmppMessage, error: Exception, client_id: str) -> None:
        logger.error("sms.send_error", log_id=getattr(smpp_message, "log_id", None), error=str(error))


class SMSClient:
    def __init__(self, config: Optional[SMSConfig] = None):
        self.config = config or SMSConfig()
        self._delivery_store: dict = {}
        self._esme: Optional[ESME] = None
        self._task: Optional[asyncio.Task] = None

    async def initialize(self, inbound_handler: Optional[Callable] = None) -> None:
        hook = HollerHook(self._delivery_store, inbound_handler)
        self._esme = ESME(
            smsc_host=self.config.smsc_host,
            smsc_port=self.config.smsc_port,
            system_id=self.config.system_id,
            password=self.config.password,
            hook=hook,
        )
        self._task = asyncio.create_task(self._esme.start())
        logger.info("sms.initialized", host=self.config.smsc_host)

    async def send(self, destination: str, message: str, log_id: str) -> None:
        msg = SubmitSm(
            short_message=message,
            source=PhoneNumber(self.config.source_address),
            destination=PhoneNumber(destination),
            log_id=log_id,
        )
        await self._esme.broker.enqueue(msg)
        self._delivery_store[log_id] = "queued"

    def get_status(self, log_id: str) -> str:
        return self._delivery_store.get(log_id, "unknown")

    async def stop(self) -> None:
        if self._esme:
            await self._esme.stop()
```

### Tool Executor Dispatch Pattern

```python
# holler/core/agent/executor.py
# Dispatches tool calls from LLM to actual Holler actions
import json
from typing import Any, Dict

class ToolExecutor:
    def __init__(self, esl, sms_client, compliance_gateway, pool):
        self._esl = esl
        self._sms = sms_client
        self._gateway = compliance_gateway
        self._pool = pool

    async def execute(self, tool_name: str, arguments: Dict[str, Any], session) -> dict:
        try:
            if tool_name == "call":
                return await self._execute_call(arguments, session)
            elif tool_name == "sms":
                return await self._execute_sms(arguments, session)
            elif tool_name == "hangup":
                return await self._execute_hangup(arguments, session)
            elif tool_name == "transfer":
                return await self._execute_transfer(arguments, session)
            else:
                return {"status": "error", "reason": f"unknown_tool: {tool_name}"}
        except ComplianceBlockError as e:
            return {"status": "blocked", "reason": str(e)}  # D-02
        except Exception as e:
            return {"status": "error", "reason": str(e)}
```

## Project Constraints (from CLAUDE.md)

These directives are binding and override any contrary research findings:

- **Language**: Python core (orchestration, agent interface) + C/Rust for latency-critical voice pipeline. Phase 3 is Python only (no C/Rust components).
- **No vendor accounts**: Core must function with zero external service accounts. SMPP trunk is the single external dependency for SMS (parallel to SIP trunk for voice).
- **Latency**: Full voice loop must complete in <800ms. Tool call interception in the pipeline must not blow this budget — filler TTS required for tool calls that take >100ms.
- **Onboarding**: Install → init → trunk → call — four commands maximum. Phase 3 must satisfy this end-to-end.
- **Apache 2.0 license**: All new dependencies must be Apache 2.0 or compatible. aiosmpplib (MIT), click (BSD), huggingface_hub (Apache 2.0), python-dotenv (BSD) — all compatible.
- **GSD Workflow Enforcement**: All file changes go through GSD execute-phase; no direct edits outside workflow.
- **Asyncio throughout**: All new components must be async-compatible. No synchronous SMPP (smpplib). No gevent.
- **Deferred initialization**: Models loaded on startup via `initialize()`, not in `__init__`. SMSClient must follow this pattern.
- **ComplianceGateway as mandatory gate**: SMS send must go through compliance before SMPP transmission. Structurally non-bypassable.
- **TYPE_CHECKING guards**: Optional imports (aiosmpplib, anthropic) must use `TYPE_CHECKING` guard pattern.
